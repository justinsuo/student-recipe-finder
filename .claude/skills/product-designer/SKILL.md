# Product Designer

Use this BEFORE you touch markup. Bad design decisions made in code are 10× harder to undo than ones made on paper.

## When to use

- New surface (page, hub, modal, panel)
- Re-laying out an existing page
- Choosing between competing CTAs
- Defining empty / error / loading state copy + visuals
- Pruning a cluttered screen

## Procedure

### Step 1 — Frame the user

Answer in 3-5 bullets before reading any code:

- Who is on this surface? (e.g. first-time student, returning user with a pantry, AI Chef power user)
- What did they just do? What page did they come from?
- What are they trying to do *right now*?
- What is the single **next best action**?
- What can be hidden, collapsed, or deferred to a secondary screen?

### Step 2 — Inspect what exists

- `src/app/page.tsx` — current home funnel
- `src/components/layout/Navbar.tsx` — what's promoted in the nav
- The relevant `src/app/<route>/page.tsx`
- `src/components/recipe/RecipeCard.tsx` and `src/components/ui/*` — reuse before invent

### Step 3 — Build an information hierarchy

Rank elements 1 (visually dominant) → 5 (de-emphasized). On any surface there should be:

- 1× hero / headline
- 1-2× primary CTA
- 3-5× supporting cards or links
- 0-1× ambient details (badges, trust signals, counts)

If your draft has 4+ "primary" things, you don't have a primary. Cut.

### Step 4 — Empty / error / loading

For every dynamic surface, write the copy and visual for:

- **Empty:** first-time, no data. Should *teach* the next action. Use `<EmptyState>`.
- **Loading:** use `<SkeletonRecipeGrid>` or a labeled spinner. Don't show a blank.
- **Error:** plain English, recovery action. No raw error text in the user UI.

### Step 5 — Trust signals (when relevant)

- "$X/serving" derived from real ingredient catalog → say it
- "Generated by AI" → mark it; never hide it
- "We don't store your ZIP" → say it on the location picker

### Step 6 — Self-critique

Ask these out loud before coding:

- "If I removed the top 30% of this screen, what would the user lose?"
- "What's the second action a user might want? Is it discoverable in ≤ 1 tap?"
- "Does this screen survive at 360px wide?"

## Files to inspect

- `src/app/page.tsx`
- `src/app/<your-target-route>/page.tsx`
- `src/components/layout/Navbar.tsx`
- `src/components/ui/EmptyState.tsx`
- `src/components/ui/SkeletonRecipeCard.tsx`

## What good output looks like

- A 1-paragraph design rationale at the top of your PR / commit body
- A short bulleted plan for the hierarchy, CTAs, and empty/loading/error
- Re-use of existing primitives, not new one-off components

## Quality checklist

- [ ] One clear primary CTA on the surface
- [ ] Hero answers "what is this page?" in ≤ 8 words
- [ ] Empty state teaches the next action, not just "No results"
- [ ] Loading state preserves layout (skeleton, not spinner-in-void)
- [ ] Error state has a recovery action
- [ ] At 360px width, no horizontal scroll, no wrap on the header
- [ ] Nothing developer-only is visible (model IDs, JSON, "TODO")

## Common mistakes

- Adding a 4th CTA to "give users more options" — fewer is more
- Putting two emerald solid buttons next to each other (only one is primary)
- Generic "No results found." — always provide a path forward
- Burying the most-used action below the fold
- Building a new component when `Button` / `Card` / `Badge` already work
